# Web app

The static site: a Spec Explorer over the current WSDOT Standard Specifications, and a
Section History viewer that shows each section's 26-year life across every edition.

Vite + React + TypeScript + Tailwind. No backend — every screen is served as static
files, and the section data is fetched lazily by division.

## Develop

```bash
# From the repo root, generate the data the app reads (fetches 17 PDFs, ~5 min once):
make app-data

# Then, in this directory:
npm install
npm run dev      # local dev server
npm test         # unit + component tests (Vitest)
npm run build    # type-check + production build into dist/
```

The app expects its data at `public/data/` (git-ignored — it's generated by
`make app-data`, not vendored). A fresh clone must run that once before `npm run dev`.

## How it's organized

```
src/
  config.ts            product name + the unofficial disclaimer, in one place
  types.ts             the shapes of the JSON emitted by pipeline/build_app_data.py
  lib/
    api.ts             lazy, cached fetches of index / per-division text / history
    sectionNumber.ts   book-order comparison (mirrors the Python parser's sort key)
    indexContext.tsx   loads the ~250 KB index once, shares it by context
    format.ts          paragraph splitting, event wording, churn phrasing
    useAsync.ts        load-with-cancellation hook
  components/
    Timeline.tsx       the centerpiece: a section's history as a vertical timeline
    DiffView.tsx       renders a revision's precomputed word-level diff as a redline
    SearchBox.tsx      instant client-side keyword search over the index
    ...
  routes/
    Home / Browse / SectionPage / HowBuilt / About / NotFound
```

Routing is hash-based (`#/section/1-07.1`) so deep links and refreshes work on GitHub
Pages and file shares with no server rewrites. Assets use a relative base, so the same
build runs at a domain root or a project subpath.
